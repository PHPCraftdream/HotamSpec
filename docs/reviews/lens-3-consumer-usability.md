# Линза 3 — Юзабельность Hotam-Spec для нового потребителя «с улицы»

## 1. Первый контакт (README)

`README.md` — хороший, честный документ: есть elevator pitch, диаграмма tension graph, таблицы концепций и инструментов, и даже секция «Adopt: your domain in 15 minutes». Но:

- **README не знает о портативности.** Единственный путь установки — `git clone` + `cd spec && uv sync`. Ни слова про `pip install hotam-spec`, ни одного упоминания 29 CLI entry points (`hotam-create-domain`, `hotam-what-now`, …). Вся секция Adopt написана через `uv run python tools/create_domain.py` из каталога `spec/` внутри клонированного репо фреймворка. Потребитель, которому поставили пакет как зависимость, из README не узнает, что ему делать: сценарий «работать в СВОЁМ репо» документально не существует.
- Ни один .md во всём репо (кроме автогенерированного внутреннего `spec/docs/thinking/graph.md`) не упоминает `HOTAM_SPEC_PROJECT_ROOT`, `hotam-*`-команды или маркер `.hotam-spec-project`. Портативность документирована только в docstring `spec/src/hotam_spec/project_paths.py`.
- Бейдж «944 passing» — захардкоженное число, быстро устареет.

## 2. Первая команда

Логичная первая команда для консюмера — `hotam-setup-hooks` + `hotam-create-domain … --activate`, но эта последовательность **нигде не описана в CLI-форме**. Пошаговый рецепт Adopt в README фактически рабочий (stakeholders → axis → requirements → conflict → gen-spec → what-now — это хороший минимальный сценарий!), но:

- он привязан к самохостящейся форме (`uv run python tools/...` из `spec/`);
- шаг 4 требует руками писать JSON через `echo '{"kind":"Stakeholder",...}'` — формат полей нигде не задокументирован для человека, кроме исходника `spec/src/hotam_spec/proposal.py`; ошибка в одном поле = чтение кода;
- непонятно, обязательны ли hooks (`setup_hooks.py`) для консюмера, который вообще не использует Claude Code — README подаёт их как обязательный шаг 2, хотя это чисто AI-операторная инфраструктура;
- `docs/demo/CONSTITUTION.md` — устаревший артефакт (упоминает «Tensio», `spec/content/graph.py` — пути, которых больше нет). Как «пример для новичка» он вреден: демонстрирует прежнее имя проекта и мёртвую структуру.

## 3. Ошибки

`ProjectRootUnresolved` (`spec/src/hotam_spec/project_paths.py:233-273`) — редкий пример **хорошей** диагностики: перечисляет все R1–R6 источники, состояние env-переменных, CWD, и заканчивается actionable-инструкцией («Set one of: env HOTAM_SPEC_PROJECT_ROOT=…, create a marker…»). Замечания:

- метки «R1…R6» — внутренний жаргон резолвера, для стороннего человека это шум (хоть и безвредный);
- нюанс R6: в fallback-цепочке `project_root()` почти никогда не вернёт None (repo_root() фреймворка обычно есть), значит реальный консюмер скорее получит не исключение, а **тихо неправильный корень** — данные запишутся в site-packages/каталог установки фреймворка. Это хуже понятной ошибки и smoke-тестами через tmp_path не ловится;
- R3-маркеры очень широкие: любой репозиторий с `CLAUDE.md` или каталогом `tickets/` будет молча признан hotam-проектом — ложные срабатывания у консюмера, использующего Claude Code, гарантированы.

## 4. Терминология

Разделение аудиторий соблюдено правильно: корневой `CLAUDE.md` — кристалл для AI-оператора (и явно так устроен), человеку читать его не нужно; для человека есть README + `docs/methodology/README.md` (хорошо написанный философский текст). Но словарь — серьёзный барьер:

- **Осмысленны без обучения:** Requirement, Conflict, Assumption, DRAFT/OPEN — интуитивно понятны.
- **Требуют объяснения, но короткого:** steward (≠ owner — ключевая и красивая идея, объясняется одним абзацем), axis, SETTLED.
- **Чистая внутренняя философия:** «tension graph», «mediation loop», «operator crystal», «substrate», «sensorium», «cipher», «crystallize before split», ENFORCED/STRUCTURAL/PROSE, «anti-relitigation». В README Adopt-секции уже проскакивают «operator crystal» и «pulse» без определения. Новичок не отличит, что из этого обязательная модель данных, а что — метафизика AI-самомоделирования, нужная только домену `hotam-spec-self`.

## 5. Барьер входа vs выгода

**Реальная ценность для обычной команды** сводится к трём вещам: (а) конфликт требований — записанный объект с нейтральным арбитром и rationale (анти-релитигация — это реально полезно); (б) `what_now` — детерминированный «что дальше»; (в) lifecycle допущений — протухшие допущения подсвечивают зависимые решения. Для этого достаточно **6 концепций**: Stakeholder, Requirement, Axis, Conflict, steward≠owner, Assumption.

**Цена сейчас:** пакет тащит с собой всю мета-онтологию (Operator, budgets, crystals, spawn-логи, hooks, tickets, delegations, 29 CLI-команд без единой consumer-документации), и документация не отделяет «starter path» от «self-hosting core». Честная оценка: мотивированный разработчик доберётся до первого Conflict-node за ~1-2 часа (благодаря Adopt-секции), но окупаемость наступит только если команда реально ведёт спорные решения; при этом риск бросить на этапе «зачем мне stewards, JSON-пропозалы и hooks ради трёх требований» высок.

## Конкретные рекомендации

1. **Consumer README / QUICKSTART-CONSUMER.md**: путь `pip install hotam-spec` → `hotam-create-domain my-shop --activate` → `hotam-what-now` в ЧУЖОМ репо, с явным объяснением как резолвится project root (`.hotam-spec-project` / `[tool.hotam-spec]`). Сейчас этого документа нет вообще — это пробел №1.
2. **Реальный e2e-тест «с улицы»**: `pip install` в чистый venv, пустой чужой каталог, прогон quickstart. Особо проверить R6-fallback: консюмер без маркеров не должен молча писать в каталог установки фреймворка — лучше падать с диагностикой, чем угадывать.
3. **Справочник Proposal JSON для человека** (одна страница: kinds + обязательные поля + примеры), чтобы не читать `proposal.py`. Либо интерактивные `hotam-add-requirement`/`hotam-add-stakeholder`, убирающие ручной JSON из starter path.
4. **Пометить hooks/sensorium как опциональный AI-слой** в Adopt-секции: шаги для «команда без Claude» и «команда с AI-оператором» — раздельно.
5. **Удалить/обновить `docs/demo/`** — устаревший (имя Tensio, мёртвые пути), вводит новичка в заблуждение.
6. **Глоссарий из 6 стартовых терминов** в README + явная граница: «всё про operators/crystals/budgets нужно только если вы сажаете AI-оператора».
7. Заменить захардкоженный бейдж числа тестов; сузить R3-маркеры (`CLAUDE.md` и `tickets/` — слишком общие) или требовать `.hotam-spec-project`/`[tool.hotam-spec]` как единственные явные маркеры для consumer-режима.

Ключевые файлы: `README.md`, `spec/pyproject.toml`, `spec/src/hotam_spec/project_paths.py`, `docs/methodology/README.md`, `CONTRIBUTING.md`, `docs/demo/` (устаревший).
