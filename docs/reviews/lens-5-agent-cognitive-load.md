# Линза 5 — Когнитивная нагрузка на агента-оператора

Честная оценка по пяти пунктам линзы, на основе чтения `CLAUDE.md` (27 559 символов), `spec/tools/what_now.py`, `spec/tools/attention_hook.py`, живого запуска `spec/tools/attention.py` и git-истории.

## 1. CLAUDE.md как «пробуждение»: наполовину кристалл, наполовину оглавление

Что работает: LIVE-STATE даёт top action / debt / бюджет — прямо сейчас видно, что делать (`R-decided-by-verifiable-signature`, P4). Мед.петля с командами — операционна. Domain Map, Recently-Rejected — самодостаточны.

Что не работает без дочтения:
- Секция «Methodology — how to think» — это 30 обрывков первых предложений докстрингов, часть из которых обрезана бессмысленно («RULE: used by check_typed_anchors to verify GOAL- prefix discipline...» — не правило, а справка о реализации). Чтобы реально следовать §Conflict/§Proposal, надо открыть `spec/docs/thinking/*.md`.
- Constitution index — только идентификаторы без claims. Чтобы узнать, что запрещает `R-machine-check-syntactic`, надо лезть в `REQUIREMENTS.md` (мегафайл).
- Tool reference — однострочники (см. п.4).

Вердикт: для **простого следующего шага** crystal самодостаточен (это его сильная сторона — «being lost structurally impossible» реально достигнуто). Для **любой содержательной работы** (написать proposal, разрулить конфликт) — это указатель, требующий 3–10 дополнительных чтений. Это осознанный компромисс после «crystal diet 78.5k→25.7k» (коммит 71f8162): плотность купили ценой самодостаточности.

## 2. Мед.петля: аспирационная строгость, фактическое сжатие

Git-история показывает реальный ритм: коммиты — это «волны»/«этапы» (etap 1.1…6.3, wave 15–17), каждый из которых накрывает *пачку* изменений одним LAND. То есть по факту работают шаги 4–6 (TRANSLATE→PRESENT→LAND — они инструментально принуждены: `apply_proposal.py`, gate, closure), а шаги 1–3 (ORIENT/LOCATE/CONFRONT) — неверифицируемы и почти наверняка сжимаются:

- ORIENT автоматизирован хуком (emit_cipher) — агенту не надо ничего делать, значит шаг «выполняется» тривиально.
- CONFRONT через `tools/confront.py` — лексический overlap-ранкер; нет никакого следа его запусков в .runtime-логах, ни один коммит не упоминает confront. Сам фреймворк это признал: `R-boot-cite-measured` REJECTED как «form-metric theatre», а conduct переведён в `R-agent-conduct-is-rules-not-tests` [P] — т.е. проект **явно решил**, что дисциплина шагов 1–3 — это правила-проза, а не проверяемое поведение. Честно, но это и есть признание аспирационности.
- Для мелких запросов («покажи X», «поправь опечатку») 6 шагов — это оверхед в разы больше самой работы; выход «writing nothing = valid outcome» есть, но формально всё равно требует пройти петлю.

Вердикт: петля реально работает как **воронка для изменений графа** (там она принуждена инструментами), а не как протокол на «ANY input». Заявление «every input — six steps» — аспирация.

## 3. Constitution index: фоновый шум с двумя полезными функциями

~110 идентификаторов без текста claims. LLM не «сверяется со списком» — он не может выполнить lookup по id, которого не понимает. Реальные функции индекса:

1. **Прайминг словаря**: агент видит, что `R-no-hand-edit-graph` существует, и с большей вероятностью процитирует правильный якорь (R-speak-by-reference). Это работает — но для этого не нужны все 110, работают ~15–20 часто цитируемых.
2. **Легитимация ссылок**: не выдумывать несуществующие R-anchors.

Флаги [E]/[S]/[P] агент в момент решения не различает. По факту защита от нарушений — не индекс, а enforcer-тесты (T1/T2 gate): нарушил [E]-правило — тест упал. Значит [E]-часть индекса (~85 правил) **дублирует** то, что и так поймает gate, и держится в контексте зря. Реально нуждаются в резидентности только [P]/[S]-правила (~25) — их ничто кроме внимания агента не ловит.

## 4. Tool reference: однострочников хватает для ~60%, длинный хвост требует дочтения

Хорошие однострочники (самодостаточны): what_now, confront, gen_spec, ticket_*, apply_proposal — имя+строка = достаточный сигнал.
Плохие: `tick` («closed-loop diagnostic driver (advisory, M32 conservative)» — что такое M32? когда его запускать?), `gate` vs `gate_status` vs `closure` (три инструмента вокруг одного гейта, различие тонкое), `attention` vs `attention_hook` vs `emit_cipher` (три среза одного пульса), `audit_tensions` vs `audit_atomicity`. Плюс за кадром 29 cli/-скриптов, вообще не отражённых в crystal. Практика: агент угадывает по имени, при промахе читает `spec/docs/tools/*.md` (33 файла) — одно-два дочтения на сессию терпимо, но выбор из 35 по одной строке систематически ведёт к «запущу what_now, а дальше следую его подсказкам» — т.е. реальная навигация идёт через императивы в выводе инструментов (они содержат команды «run python tools/X.py»), а не через Tool reference. Это, кстати, правильный механизм — и он делает секцию Tool reference наполовину лишней.

## 5. Attention: 40 сигналов, 38 из них — один и тот же P0. Приоритезация инвертирована

Проверил живьём: `attention.py` выдаёт 40 сигналов — **38 идентичных** P0 REFLECTION «REJECTED ... has NO structural `replaces` edge — migrate it» (пометка «Advisory; never a gate» внутри P0!), 1 P0 enforcement-gradient, 1 P4 OPEN (единственное, что требует решения стьюарда).

Это классическая инверсия: наименее срочная механическая рутина (миграция replaced_by у давно закрытых REJECTED) занимает высший приоритет и 95% полосы, а единственный содержательный сигнал — внизу. «P0 = advisory, never a gate» — оксюморон, размывающий саму семантику P0. Агент неизбежно обучается паттерну «REFLECTION = игнорировать», после чего **настоящий** P0 (over-budget, dead-assumption-on-enforcer) тоже будет проигнорирован — habituation, тот самый эффект, ради борьбы с которым уже вводили delta-режим в attention_hook и 5%-порог revisit. Смягчение частично есть: delta-режим хука печатает одну строку при неизменном наборе. Но при *любом* изменении (закрыл один тикет) стена из 38 строк возвращается целиком.

## Предложения по сокращению (конкретно)

1. **Схлопнуть 38 REJECTED-сигналов в один агрегат** — «38 REJECTED atoms await structural replaces-edge migration (batch job) — see docs/gen/HISTORY.md». Прецедент уже есть: P5 latent-connectors рендерятся по кластерам, а не по парам. Одна строка вместо 38, шум −95%. Ещё лучше: это чисто механическая миграция — один batch-proposal закрывает все 38 навсегда.
2. **Ввести отдельный band для advisory** — «Advisory; never a gate» не может жить в P0. Либо P7 ADVISORY, либо поле severity, отделённое от priority. P0 должен снова означать «брось всё».
3. **Constitution index: оставить резидентными только [P]/[S]** (~25 правил, единственные, которые держатся вниманием агента), [E]-правила заменить одной строкой «85 ENFORCED rules guarded by gate — full index: docs/gen/CONSTITUTION.md». Экономия ~4–5k символов и, важнее, внимания.
4. **Tool reference сгруппировать по задаче, а не по алфавиту** — «Пульс: what_now/attention/emit_cipher · LAND: apply_proposal→gate→closure→gate_status · Тикеты: ticket_* · Скаффолдинг: create_*». Выбор из 6 групп когнитивно дешевле выбора из 35 строк; описания редких инструментов (tick, audit_*) можно убрать в docs целиком.
5. **Честно tierовать мед.петлю** — узаконить то, что уже происходит: «полная 6-шаговая петля — для изменений графа; для read-only запросов достаточно ORIENT+ответ с якорями». Это снимет разрыв между декларацией и практикой (тот же ход, что уже сделали с R-boot-cite-measured→conduct-is-rules).
6. **Methodology-секцию** заменить машинно-обрезанные первые предложения на написанные вручную одно-предложенные резюме (или поле summary по образцу `R-crystal-carries-short-form`, который уже сделал это для requirements — применить тот же механизм к thinking-докам). Сейчас треть строк там несёт ноль сигнала («RULE: used by check_X to verify...»).

**Итоговая оценка**: фреймворк выдающимся образом решил «being lost» (LIVE-STATE + what_now — лучшая часть) и принудил дисциплину записи (LAND-воронка). Реальная перегрузка — не объём (27k из 150k — запас огромный), а **разбавление сигнала**: 38 копий одного advisory под маркой P0, 110 нераскрытых идентификаторов и 35 однострочников конкурируют за то же внимание, что и один действительно открытый вопрос стьюарда. Диета нужна не по символам, а по числу одновременно предъявляемых различимых элементов.
