# Линза 2 — Переусложнение и возможности упрощения

## 1. Онтология: типы, которые существуют ради самой онтологии

Фактическое использование (по `domains/hotam-spec-self/graph.py`, 4216 строк, и `domains/hotam-dev/graph.py`, 345 строк):

| Тип | self | dev | Вердикт |
|---|---|---|---|
| Requirement | 270 (из них 38 REJECTED) | 9 | ядро, оправдан |
| Assumption | 14 | 2 | оправдан |
| Axis | 9 | 2 | пограничный |
| Conflict | 8 | 2 | ядро методологии, оправдан |
| Stakeholder | 4 | 2 | оправдан, но см. ниже |
| **Goal** | **1** | 0 | недоиспользован |
| **Operator** | **1** | 0 | недоиспользован |
| **Process** | **1** | 0 | недоиспользован |
| **EntityType** | **0** | 1 | практически мёртв |
| Variant | 2 | 0 | оправдан только вместе с Conflict/HELD |
| Signoff | 1 | 0 | только что добавлен, 1 живой пример |

Конкретные точки:

- **Goal / Process / Operator / EntityType — четыре типа с суммарно 4 инстансами на 291 узел.** Каждый тянет за собой: свой модуль в `src/` (goal 2 checks, process 4 checks, operator 265 строк + 2 checks, entity 75 строк + **12 checks** при нуле инстансов в основном домене), свой Proposed*-kind, свои секции в gen_spec, свою строку в Concept Map. `entity.py` + 12 check_* + `create_entity_type.py` — это инфраструктура «на вырост» для внешних доменов, которых пока нет. **Что теряем, оставляя:** каждый новый читатель обязан выучить 11 типов вместо 5; invariants.py и gen_spec.py растут на каждый тип. **Что приобретаем:** гипотетическую готовность к бизнес-доменам. **Предложение:** заморозить EntityType/Process/Goal как «extension types» (отдельный модуль, не в кристалле, не в Concept Map, не в Proposed*-перечне по умолчанию) до появления второго реального потребителя. Goal с 1 инстансом легко выражается Requirement'ом с префиксом GOAL- — собственный тип с owner-check не добавляет ничего, что не даёт `owner` у Requirement.
- **Relation-kinds:** `refines` 23, `supports` 11, `depends_on` 6, `replaces` — **0 литеральных использований в графах** (есть только ветка `if rel.kind == "replaces"` в graph.py:488; анти-релитигация реально живёт в прозе `why=("REPLACES ...")` — gen_spec парсит её маркером `_has_rejected_replaces_marker`). Т.е. типизированное ребро добавлено, а данные по-прежнему в прозе — двойной механизм для одной функции. **Предложение:** либо мигрировать REJECTED-узлы на ребро `replaces` и убрать прозо-парсер, либо убрать kind — сейчас худший вариант: оба пути живы. Разница `supports` vs `refines` нигде семантически не эксплуатируется (ни один check не различает их) — можно схлопнуть в одно ребро без потери функции.
- **Stakeholder на 4 инстансах** — по сути enum из строк; тип оправдан только check'ом resolver≠owner.

## 2. Enforcement-градиент: 3×2 матрица там, где на практике живёт булево

Реальное распределение: 198/227 ENFORCED, ~14 STRUCTURAL, ~9 PROSE, 25 INHERENTLY_PROSE, 5 «closeable debt». Плюс ортогонально status: SETTLED/DRAFT/OPEN/REJECTED (+ OPEN несёт вопрос в строке статуса — `"OPEN(<question>)"`, стрингли-типизированный статус, который потом парсится `_extract_open_question`).

- **STRUCTURAL — самая мутная ступень**: «есть структура, но не тест». На практике это просто «не ENFORCED, но мы не чувствуем вины». Различие STRUCTURAL/PROSE не эксплуатируется ни одним потребителем кроме burn-down-метра; для решения «что делать дальше» важен только один бит: enforced или нет + один бит enforceable или нет. **Предложение:** enforcement = ENFORCED | NOT_ENFORCED, отдельный флаг `inherently_prose: bool`. STRUCTURAL-узлы честно становятся NOT_ENFORCED-долгом — это даже усиливает ratchet.
- `"OPEN(<question>)"` внутри строки status — микро-переусложнение: вопрос должен быть полем (`question: str`), а не парситься регэкспом из статуса.
- Что теряем при упрощении: гранулярность burn-down-отчёта. Что приобретаем: минус одна ось, которую надо объяснять каждому агенту.

## 3. Конституция: 100+ правил — это не конституция, это кодекс

В индексе ~110 business/discipline правил + 112 «framework internals». Никакой агент не держит это в голове; на практике работают: (а) генеративный закон (1 фраза), (б) мед.петля (6 шагов), (в) `confront.py`/`gate.py` как механические стражи. Остальные правила читаются лениво через LOCATE — и это нормально, **проблема не в существовании 227 атомов, а в том, что все они выведены в индекс кристалла**, как будто их надо знать заранее.

Конкретная избыточность — целые кусты правил про один механизм:
- `R-land-gate-tier-selector` + `-fails-closed` + `R-land-tier-trace` + `-best-effort` + `-skips-dry-run` + `R-tiered-gate-not-a-commit-gate` + `R-commit-boundary-checkable` — 7 атомов про один инструмент gate.py. Атомизация (R-requirement-claim-is-atomic) применена рекурсивно к самой себе и произвела дробление ниже полезного зерна: тестируемая единица — gate.py, а не 7 отдельных утверждений.
- Аналогично кусты `R-attention-*` (4), `R-tension-audit-*` (3), `R-active-loop-*` (3).

**Предложение:** ввести уровень «правило-кластер» (один родительский атом на инструмент/механизм, дети — как enforced_by-строки, не как отдельные узлы конституции) — или хотя бы сворачивать кусты в индексе кристалла в одну строку. Сокращение индекса ~110 → ~40 строк без потери одного enforcement'а.

## 4. gen_spec.py (4786 строк): корень сцепленности

Структура: ~110 функций, из них ~15 `build_*` (по одной на артефакт docs/gen/), ~20 `_render_*` блоков CLAUDE.md, плюс domain-scan, agent-scan, concept-map-scan, tool-help-capture (запускает CLI `--help` сабпроцессами!), memoization, dirty-index доменов. 104 упоминания CLAUDE.md внутри; **147 файлов тестов ссылаются на gen_spec/CLAUDE**.

Корень — не размер, а **три архитектурных решения**:

1. **Артефакт как строка, а не как модель.** Каждый `build_*` конкатенирует markdown вручную; тесты (`test_operator_seed.py` и др.) ассертят *текст сгенерированного кристалла*. Значит любой рефакторинг рендеринга ломает десятки текстовых ассертов — файл «не поддаётся распилу» потому, что его контракт — байты вывода, а не структура. Правильно с самого начала: gen_spec строит промежуточную модель (list of typed blocks), тесты ассертят модель, рендер в markdown — тонкий тривиальный слой. Тогда каждый build_* — отдельный модуль с независимым контрактом.
2. **Требования о содержимом кристалла ENFORCED через тесты над выводом генератора** — само требование «Role-блок называет закон» превращено в тест на подстроку. Это замыкает петлю spec→gen_spec→test→spec: тронуть генератор = тронуть enforcement 30+ правил одновременно.
3. **Один процесс делает всё за раз** (все домены, все агенты, все доки, concept map, tool --help сабпроцессами). Нет границы «этот build читает только граф» — некоторые читают ФС, некоторые запускают процессы, некоторые читают выходы других build'ов (`extract_live_state_block` парсит собственный прошлый вывод!). Разрезать нельзя, потому что зависимости идут через неявное состояние ФС.

Ирония: фреймворк, чей главный тезис — «drift между слоями структурно невозможен», породил файл, где drift невозможен потому, что всё слиплось в один орган.

## 5. Cognitive tax для нового оператора (чтение CLAUDE.md глазами новичка)

Кристалл ~26.5k chars. До первого полезного действия реально нужны: генеративный закон (3 строки), мед.петля (6 шагов, ~30 строк), 3 команды (`what_now.py`, `confront.py`, `apply_proposal.py`) и «resolver решает, ты — нет». Это ~15% файла. Остальные ~85% — Constitution index (110 ссылок-идентификаторов без claims — новичку бесполезен, это lookup-таблица, а не знание), Concept Map (30 строк табличной статистики «сколько checks/tests» — не нужна для действия вообще), Tool reference (35 инструментов одной строкой — из них в первый день нужны 4), Recently rejected (нужен только на шаге CONFRONT, и там уже есть tool). Плотность жаргона высокая: «resident seed», «crystal», «sensorium», «cipher», «altitude», «hand» — внутренний диалект, требующий glossary для чтения собственного boot-файла.

**Предложение:** кристалл = закон + петля + 4 команды + LIVE-STATE (~5k chars); Constitution index, Concept Map, Tool reference — отдельные генерируемые файлы, на которые кристалл ссылается одной строкой (LOCATE-шаг и так велит идти в docs/gen/). Это одновременно решает и будущую проблему бюджета кристалла (R-budget-measure), и tax новичка.

## Сводка предложений (без потери функции)

1. Goal/Process/EntityType/Operator-тип → заморозить/вынести в extension; Goal слить с Requirement (GOAL-префикс уже есть).
2. `supports`+`refines` → одно ребро; `replaces` — либо мигрировать данные и убрать прозо-парсер, либо убрать kind (сейчас дублирование).
3. Enforcement → бинарный (ENFORCED/NOT) + флаг inherently_prose; вопрос OPEN — в поле, не в строку статуса.
4. Кусты правил одного инструмента (gate 7 шт., attention 4, tension-audit 3, active-loop 3) → правило-кластеры; индекс кристалла ~110 → ~40 строк.
5. gen_spec: контракт тестов перевести с текста на промежуточную модель блоков — это и есть недостающее условие безопасного распила; убрать `--help`-сабпроцессы (docstring уже есть в файле).
6. Кристалл сократить до закон+петля+команды+LIVE-STATE; справочные блоки — по ссылке.

Ключевые файлы: `spec/tools/gen_spec.py`, `spec/src/hotam_spec/requirement.py` (RELATION_KINDS, строки 83–94), `spec/src/hotam_spec/invariants.py` (88 check_* / 4482 строки), `spec/src/hotam_spec/proposal.py` (11 Proposed*-видов), `domains/hotam-spec-self/graph.py`, `CLAUDE.md`.
