# GEN-CODE-CONTRACT.md — контракт `hotam gen-code`

Статус: черновик этапа 1 (план: `../../PLAN-gen-code.md`), ждёт финального утверждения steward после этапа 2 (первый реальный прогон).
Область: правила, по которым движок HotamSpec порождает Go-код (модели, lifecycle-методы, тесты) из графа домена — в дополнение к уже существующей генерации `docs/gen/*.md`.

## 0. Принцип зеркала (обязателен к прочтению перед чтением остального)

Свободный текст (`claim`, `description`, `why`) непроверяем сам по себе. Тестируемость, которую
порождает `gen-code`, двухслойная:

1. **Исполнимый слой** — код реально запускается (`go build && go test`): нелегальный переход
   роняет тест, отсутствующее required-поле роняет `Validate()`, gate не пропускает не-terminal
   артефакт дальше по pipeline.
2. **Слой-зеркало** — код нарочно прямой и однозначный (1:1 со структурой требования), поэтому
   служит артефактом LLM/человеческого аудита: расхождение между текстом требования и кодом
   видно визуально, без домысливания.

Все правила ниже (mapping, идентификаторы, формализуемость) существуют, чтобы это зеркало
оставалось однозначным. Где однозначность невозможна — генератор обязан **явно и громко
отказать** (ошибка сборки/генерации), а не молча угадать.

## 1. Расположение и структура выхода

```
domains/<d>/gen/go/
  go.mod                    — отдельный Go-модуль (module hotam-gen/<d>, go <версия из движка>)
  entities.go                — structs + enum состояний + Validate() (этап 2)
  lifecycle.go                — transition-методы (этап 3)
  lifecycle_test.go           — table-driven тесты переходов (этап 3)
  requirements_test.go        — атомы требований → assertion / честная пометка (этап 4)
  requirements_audit.md       — аудит-таблица требование-атом ↔ файл:строка (этап 4)
  pipeline_test.go            — сквозной gate-тест P-G0..P-G4 (этап 5)
```

- `domains/<d>/gen/go/` — отдельный Go-модуль (`go.mod`), НЕ пакет внутри модуля движка/потребителя.
  Причина: потребительский проект (`PRAT-hotam`) не обязан быть Go-проектом; вложенный модуль
  изолирует зависимость от `go.mod` родителя и позволяет `go test ./...` внутри него независимо.
- Активируется флагом `gen_profile` в `manifest.json` домена так же, как консюмерский профиль
  документации: если `require_gen_code: true` в manifest (новое поле, аналог `require_provenance`) —
  `hotam gen-code` обязателен для land/gen-spec pipeline; иначе — доступен по явному вызову, но не
  блокирует. Значение по умолчанию — `false` (opt-in, как provenance-гейт, чтобы `hotam-spec-self`/
  `hotam-dev` не были задеты без явного решения).
- Ownership-маркер — как в `docs/gen/*.md`: первая строка каждого сгенерированного `.go`-файла —
  `// AUTOGENERATED from the domain graph.json — do not edit by hand. Edits: methodology/graph -> hotam gen-code`.
  Ручные правки внутри `domains/<d>/gen/go/` запрещены (симметрично правилу для `docs/gen/`).

## 2. Mapping EntityType → Go

| Элемент графа | Go-конструкция |
|---|---|
| `EntityType.slug` | имя struct (PascalCase от переведённого/транслитерированного slug, §4) |
| `EntityType.fields[]` | поля struct; `kind: string` → `string`; `kind: enum` → отдельный `type <Field>Kind string` с константами |
| `field.required: true` | ветка в `Validate() error` — `if x.<Field> == "" { return fmt.Errorf(...) }` |
| `field.ref_target` (не пусто) | комментарий-ссылка + место для будущей inter-entity проверки (реальная проверка — если `ref_target` резолвится к другому EntityType домена, иначе честная TODO-пометка, не имитация) |
| `lifecycle.states[]` | `type <Entity>State string` + константы; `kind: initial` → значение по умолчанию конструктора; `kind: terminal` → состояние без исходящих переходов в transition-таблице |
| `lifecycle.transitions[]` | по одному методу `func (x *<Entity>) <Event>() error` на `event`; тело проверяет `x.State`, при несовпадении — ошибка; при совпадении — переход + (если задан) hook для `guard`/`guard_assumption` |
| `lifecycle.transitions[].why` | комментарий над методом (дословно, не перефразируется) |
| `EntityType.why` | doc-comment над struct (связь с породившим `R-gate-*`) |

Все графовые поля, для которых нет строки в этой таблице на момент генерации (новый `kind`,
новый элемент `lifecycle`) — генератор обязан явно отказать с именем неизвестного элемента,
не пропускать молча.

## 3. Классификация атомов требования (для этапа 4)

Дополняет таксономию из `PLAN-gen-code.md`. Перед генерацией `requirements_test.go` каждое
SETTLED-требование домена прогоняется через следующую эвристику формализуемости:

| Признак в claim | Тип атома | Зеркало |
|---|---|---|
| claim ссылается на `EntityType.field` (по имени, включая переведённое) | сущность/поле | assertion в `Validate()`-тесте |
| claim упоминает пару состояний из `lifecycle.states` конкретного EntityType | состояние/переход | assertion в `lifecycle_test.go`-стиле (тот же файл, что этап 3, но триггер — текст требования, а не сам граф lifecycle) |
| claim содержит мета-токен (`MUST`/`ALWAYS`/`NEVER`/`ONLY`/`ANY`/`MUST NOT`, см. `internal/generator/claudemd_static.go`) И типизированный якорь (`R-gate-*`, `EntityType.slug`, `id` артефакта) | гейт/порядок | assertion-предикат, зеркалящий именно этот якорь |
| claim описывает связь между двумя разными EntityType/доменами | инвариант между сущностями | assertion, если оба конца резолвятся в графе; иначе — честная пометка (§0) |
| ничего из вышеперечисленного не найдено (чистое процессное/ролевое намерение, напр. «AI presents, steward decides») | без структурного носителя | **честная пометка**: `t.Log("no structural atom — see claim: <дословный текст>")`, требование ВСЁ РАВНО получает именованный `func Test_<ID>` (виден в `go test -v`), но без `t.Skip` — тест проходит, будучи честно пуст по существу, а не притворяется skip'нутым по техническим причинам |

Эвристика детерминирована (без LLM-угадывания на этапе генерации) — LLM участвует только в
аудите результата (§0), не в самой классификации. Если эвристика в будущем ошибается на
конкретном требовании — это баг генератора, чинится добавлением правила в эту таблицу, не
хардкодом исключения для одного id.

## 4. Идентификаторы: перевод + транслитерация аббревиатур (решение steward, 2026-07-15)

Граф несёт русские имена (`ид_фт`, `черновик`, `утвердить-pm`). Правило превращения в Go-идентификатор:

1. Составное имя разбивается на части по разделителям (`_`, `-`).
2. Каждая часть ищется **сначала** в таблице аббревиатур (§4.2) — если найдена, транслитерируется
   буква-в-букву в верхний регистр латиницы (не переводится).
3. Иначе часть ищется в глоссарии переводов (§4.1) — если найдена, заменяется английским эквивалентом.
4. Части, уже являющиеся латиницей/ASCII (`ac`, `api_dto`, `cosmic`, `gate`, `pm`, `v1`) — проходят
   как есть (passthrough), приводятся к общей конвенции регистра.
5. Части склеиваются в PascalCase (для экспортируемых имён) или camelCase (для методов/локальных
   переменных) в **исходном порядке слов** — без реордеринга (переупорядочивание нарушало бы
   принцип зеркала: человек/LLM должен узнавать структуру исходного русского имени в переводе).
6. **Часть, не найденная ни в одной таблице — жёсткая ошибка генерации** с точным именем
   нераспознанной части и путём до места в графе. Не тихий фоллбэк на транслитерацию, не
   угадывание перевода. Оба словаря — часть этого контракта, расширяются осознанным PR/коммитом,
   не автогенерацией на лету.

## 5. Инварианты генерации (обязательная приёмка каждого этапа 2-6)

- **Детерминизм**: два прогона `hotam gen-code` на неизменном графе дают byte-identical файлы
  (sha256-сверка, не `git diff`).
- **Идемпотентность**: повторный прогон без изменения графа не меняет ни один файл.
- **Компилируемость и проходимость**: `go build ./... && go test ./...` внутри
  `domains/<d>/gen/go/` — зелёные на любом валидном домене.
- **Мутационная проверка (этапы 3, 5)**: искусственная порча зеркала (лишний легальный переход,
  gate без проверки terminal) обязана ронять сгенерированный тест — тест, который не может
  покраснеть, не защита.
- **Ownership**: файлы домена, не движка — коммитятся в проект потребителя (`PRAT-hotam` и т.п.),
  не в `HotamSpec`.

  Ужесточение до ошибки — отдельное решение steward после накопления опыта.
- Расширение глоссария/таблицы аббревиатур для домена `gpsm-sm` (другой словарь) — будет собран
  аналогично при первом прогоне `gen-code` на этом домене (этап 7), не выдумывается заранее.
